### 设计模式part

* #### 面向对象六大原则
  
  1. Single Responsibility Principle: 单一职责原则
     * 一个类只承担一个功能
       * 降低耦合性
       * 增强可读性与可维护性
       * 增加复用性
  2. Open/Close Principle: 开闭原则
     * 对扩展开放，对修改封闭
     * 子类重写父类代码
  3. Liskov Substitution Principle: 里氏替换原则
     * 继承逻辑正确性的表象：所有父类出现的地方，都可以用子类对象来替代
  4. Interface Segregation Principle: 接口隔离原则
     * 拆分接口
  5. Dependency Inversion Principle: 依赖倒置原则
     * 高层次的类不应该依赖低层次的类
       * 例如网络通信：游戏业务逻辑依赖TCP协议，在后期如果需要换成UDP类的接口。在设计之初游戏业务本身就该调用network这个类中的send/recv
  6. Law of Demeter: 迪米特法则
     * 最少依赖原则
     * 避免单个对象变化影响其他对象
       * 增强系统稳定性
       * 降低耦合度

* ##### 工厂方法模式

* 将类初始化/创建的工作交给另一个类

    * 封装复杂的对象创建逻辑
      * 工厂内部屏蔽对象创建与初始化的复杂操作

    * 遵循开闭原则与单一之策原则
      * 提高可维护性

* ##### 原型设计模式

* 一种创建型设计模式，通过复制现有对象来创建新对象，而非实例化

* 适用与频繁创建相似对象的场景

  * 要素：

    * Prototype抽象类型：声明克隆自身的接口

      * 

      ```c#
      // 抽象原型接口
      public interface IEnemyPrototype
      {
          IEnemyPrototype Clone();
          string GetInfo();
      }
      ```

    * 具体类型：实现克隆操作的具体类

      * 

      ```c#
      // 具体原型类 - 法师
      public class Mage : IEnemyPrototype
      {
          public string Name { get; set; }
          public int Mana { get; set; }
          public int SpellPower { get; set; }
          public string Element { get; set; }
      
          public Mage(string name, int mana, int spellPower, string element)
          {
              Name = name;
              Mana = mana;
              SpellPower = spellPower;
              Element = element;
          }
      
          public IEnemyPrototype Clone()
          {
              return new Mage(Name, Mana, SpellPower, Element);
          }
      
          public string GetInfo()
          {
              return $"法师: {Name}, 法力值: {Mana}, 法术强度: {SpellPower}, 元素: {Element}";
          }
      }
      ```

    * 创建工厂

      * ```c#
        // 敌人工厂（使用原型模式）
        public class EnemyFactory
        {
            private Dictionary<string, IEnemyPrototype> _prototypes = new Dictionary<string, IEnemyPrototype>();
        
            public EnemyFactory()
            {
                // 初始化原型对象
                _prototypes["mage"] = new Mage("基础法师", 200, 30, "火焰");
            }
        
            public IEnemyPrototype CreateEnemy(string type)
            {
                if (_prototypes.ContainsKey(type))
                {
                    return _prototypes[type].Clone();
                }
                throw new ArgumentException($"未知的敌人类型: {type}");
            }
        }
        
        // 使用示例
        class Program
        {
            static void Main()
            {
                EnemyFactory factory = new EnemyFactory();
                
                // 快速创建多个相同类型的敌人
                IEnemyPrototype mage1 = factory.CreateEnemy("mage");
                
                Console.WriteLine(mage1.GetInfo());   // 法师: 基础法师, 法力值: 200, 法术强度: 30, 元素: 火焰
            }
        } 
        ```

* ##### 单例模式

* 所有的管理器都应该使用单例模式

* 日志管理器，数据库连接池等创建耗时，使用时消耗资源过多

* 饿汉模式

  * 在类加载时就使用静态成员变量创建了自己的实例

  * ```c++
    class Singleton {
        private:
        	//静态成员变量，实例化在类加载时进行
        	static Singleton instance;
        	
        	//私有化构造函数，禁止外部实例化
       		Singleton() {
                
            }
        
        	//禁止拷贝构造和赋值操作
        	Singelton(const Singleton&) = delete;
        	Singleton& operator=(const Singleton&) = delete;
        
        public:
        	//提供一个静态方法来访问实例
        	static Singleton& getInstance() {
                return instance;
            }
        
        	//其他方法......
    }
    ```

* 懒汉模式

  * 在类加载时，单例对象为空，在第一次调用时，类才会被创建

  * ```c++
    static Singleton* getInstance() {
    	if (instance == nullptr) {
            instance = new Singleton();
        }            
        return instance;
    }
    ```

  * 更加节省资源占用
  
  * 但是由于是动态创建的资源实例，在多线程环境下有安全隐患
  
    * 可能有两个实例同时创建
    * 需要加锁避免
  
* 单例系统的生命周期都应该是静态变量的生命周期（从创建起占据整个程序的生命周期）


  * 故而对单例系统额外提供的资源释放方法是不必要且累赘的

### 游戏中的状态机&&行为树

##### 状态机

* 状态驱动：有明确的当前状态和转换
* 确定性：状态转换基于明确的触发条件
* 简单直观
* 内存效率：只维护当前状态信息
* 适合线性流程：处理有明确阶段性的行为

```typescript
// 怪物状态枚举
enum MonsterState {
    IDLE = 'idle',
    PATROL = 'patrol',
    CHASE = 'chase',
    ATTACK = 'attack',
    DEAD = 'dead'
}

// 怪物状态机类
class MonsterStateMachine {
    private currentState: MonsterState = MonsterState.IDLE;
    private target: any = null;
    private health: number = 100;

    update(deltaTime: number): void {
        switch (this.currentState) {
            case MonsterState.IDLE:
                this.handleIdleState(deltaTime);
                break;
            case MonsterState.PATROL:
                this.handlePatrolState(deltaTime);
                break;
            case MonsterState.CHASE:
                this.handleChaseState(deltaTime);
                break;
            case MonsterState.ATTACK:
                this.handleAttackState(deltaTime);
                break;
            case MonsterState.DEAD:
                this.handleDeadState(deltaTime);
                break;
        }
    }

    private handleIdleState(deltaTime: number): void {
        // 空闲状态逻辑
        if (this.detectPlayer()) {
            this.transitionTo(MonsterState.CHASE);
        } else if (this.shouldPatrol()) {
            this.transitionTo(MonsterState.PATROL);
        }
    }

    private handlePatrolState(deltaTime: number): void {
        // 巡逻状态逻辑
        if (this.detectPlayer()) {
            this.transitionTo(MonsterState.CHASE);
        } else if (this.reachedPatrolPoint()) {
            this.setNextPatrolPoint();
        }
    }

    private handleChaseState(deltaTime: number): void {
        // 追逐状态逻辑
        if (!this.targetInRange()) {
            this.transitionTo(MonsterState.IDLE);
        } else if (this.canAttack()) {
            this.transitionTo(MonsterState.ATTACK);
        }
    }

    private handleAttackState(deltaTime: number): void {
        // 攻击状态逻辑
        if (!this.targetInRange()) {
            this.transitionTo(MonsterState.CHASE);
        } else if (this.health <= 0) {
            this.transitionTo(MonsterState.DEAD);
        }
    }

    private handleDeadState(deltaTime: number): void {
        // 死亡状态逻辑 - 通常播放死亡动画后销毁
    }

    private transitionTo(newState: MonsterState): void {
        console.log(`怪物状态转换: ${this.currentState} -> ${newState}`);
        this.currentState = newState;
        
        // 执行状态进入时的初始化
        this.onStateEnter(newState);
    }

    private onStateEnter(state: MonsterState): void {
        switch (state) {
            case MonsterState.ATTACK:
                this.startAttackAnimation();
                break;
            case MonsterState.DEAD:
                this.playDeathAnimation();
                break;
        }
    }

    // 辅助方法
    private detectPlayer(): boolean {
        // 检测玩家是否在视野内
        return Math.random() > 0.7; // 模拟检测逻辑
    }

    private shouldPatrol(): boolean {
        return Math.random() > 0.5;
    }

    private targetInRange(): boolean {
        return this.target !== null;
    }

    private canAttack(): boolean {
        return this.targetInRange() && Math.random() > 0.3;
    }

    private reachedPatrolPoint(): boolean {
        return Math.random() > 0.8;
    }

    private setNextPatrolPoint(): void {
        // 设置下一个巡逻点
    }

    private startAttackAnimation(): void {
        // 开始攻击动画
    }

    private playDeathAnimation(): void {
        // 播放死亡动画
    }
}

// 使用示例
const monster = new MonsterStateMachine();
setInterval(() => monster.update(0.016), 16); // 约60FPS

##### 行为树

* 树状结构：由节点组成的层次结构

* 组合模式：通过组合器节点控制执行流程

  * 

    | 节点类型      | 功能描述 | 执行逻辑               |
    | ------------- | -------- | ---------------------- |
    | **Selector**  | 选择器   | 执行第一个成功的子节点 |
    | **Sequence**  | 序列器   | 按顺序执行所有子节点   |
    | **Condition** | 条件节点 | 检查条件是否满足       |
    | **Action**    | 动作节点 | 执行具体行为动作       |
    | **Parallel**  | 并行节点 | 同时执行所有子节点     |
    | **Inverter**  | 反转器   | 反转子节点的执行结果   |

* 可复用性：节点可以重复使用，便于模块化设计
* 响应式：每个tick都会重新评估条件
* 适合复杂场景

```typescript
// 行为树节点基类
abstract class BTNode {
    abstract execute(): BTStatus;
}

// 节点执行状态
enum BTStatus {
    SUCCESS = 'success',
    FAILURE = 'failure',
    RUNNING = 'running'
}

// 选择器节点：执行第一个成功的子节点
class Selector extends BTNode {
    private children: BTNode[];
    
    constructor(children: BTNode[]) {
        super();
        this.children = children;
    }
    
    execute(): BTStatus {
        for (const child of this.children) {
            const status = child.execute();
            if (status === BTStatus.SUCCESS || status === BTStatus.RUNNING) {
                return status;
            }
        }
        return BTStatus.FAILURE;
    }
}

// 序列节点：按顺序执行所有子节点
class Sequence extends BTNode {
    private children: BTNode[];
    
    constructor(children: BTNode[]) {
        super();
        this.children = children;
    }
    
    execute(): BTStatus {
        for (const child of this.children) {
            const status = child.execute();
            if (status === BTStatus.FAILURE) {
                return BTStatus.FAILURE;
            }
            if (status === BTStatus.RUNNING) {
                return BTStatus.RUNNING;
            }
        }
        return BTStatus.SUCCESS;
    }
}

// 条件节点：检查条件是否满足
class Condition extends BTNode {
    private condition: () => boolean;
    private name: string;
    
    constructor(name: string, condition: () => boolean) {
        super();
        this.name = name;
        this.condition = condition;
    }
    
    execute(): BTStatus {
        const result = this.condition();
        console.log(`条件检查 [${this.name}]: ${result}`);
        return result ? BTStatus.SUCCESS : BTStatus.FAILURE;
    }
}

// 动作节点：执行具体行为
class Action extends BTNode {
    private action: () => BTStatus;
    private name: string;
    
    constructor(name: string, action: () => BTStatus) {
        super();
        this.name = name;
        this.action = action;
    }
    
    execute(): BTStatus {
        console.log(`执行动作 [${this.name}]`);
        return this.action();
    }
}

// 并行节点：同时执行所有子节点
class Parallel extends BTNode {
    private children: BTNode[];
    
    constructor(children: BTNode[]) {
        super();
        this.children = children;
    }
    
    execute(): BTStatus {
        let successCount = 0;
        let failureCount = 0;
        
        for (const child of this.children) {
            const status = child.execute();
            if (status === BTStatus.SUCCESS) successCount++;
            if (status === BTStatus.FAILURE) failureCount++;
        }
        
        if (failureCount > 0) return BTStatus.FAILURE;
        if (successCount === this.children.length) return BTStatus.SUCCESS;
        return BTStatus.RUNNING;
    }
}

// 装饰器节点：修饰子节点的行为
class Inverter extends BTNode {
    private child: BTNode;
    
    constructor(child: BTNode) {
        super();
        this.child = child;
    }
    
    execute(): BTStatus {
        const status = this.child.execute();
        switch (status) {
            case BTStatus.SUCCESS: return BTStatus.FAILURE;
            case BTStatus.FAILURE: return BTStatus.SUCCESS;
            default: return status;
        }
    }
}

// 怪物行为树类
class MonsterBehaviorTree {
    private root: BTNode;
    private health: number = 100;
    private playerInSight: boolean = false;
    private playerInRange: boolean = false;
    private patrolPointReached: boolean = false;
    
    constructor() {
        this.buildBehaviorTree();
    }
    
    private buildBehaviorTree(): void {
        // 构建完整的行为树结构
        this.root = new Selector([
            // 优先级1: 死亡检查
            new Sequence([
                new Condition("是否死亡", () => this.health <= 0),
                new Action("死亡行为", () => {
                    console.log("💀 怪物死亡");
                    return BTStatus.SUCCESS;
                })
            ]),
            
            // 优先级2: 攻击玩家
            new Sequence([
                new Condition("玩家在攻击范围内", () => this.playerInRange),
                new Action("攻击玩家", () => {
                    console.log("⚔️ 攻击玩家");
                    return BTStatus.SUCCESS;
                })
            ]),
            
            // 优先级3: 追逐玩家
            new Sequence([
                new Condition("发现玩家", () => this.playerInSight),
                new Action("追逐玩家", () => {
                    console.log("🏃 追逐玩家");
                    return BTStatus.RUNNING;
                })
            ]),
            
            // 优先级4: 巡逻
            new Sequence([
                new Condition("需要巡逻", () => !this.patrolPointReached),
                new Action("巡逻", () => {
                    console.log("🚶 巡逻中");
                    this.patrolPointReached = Math.random() > 0.7;
                    return this.patrolPointReached ? BTStatus.SUCCESS : BTStatus.RUNNING;
                })
            ]),
            
            // 优先级5: 空闲状态（默认行为）
            new Action("空闲", () => {
                console.log("😴 空闲中");
                return BTStatus.SUCCESS;
            })
        ]);
    }
    
    // 更新行为树
    update(): void {
        // 模拟环境变化
        this.simulateEnvironmentChanges();
        
        // 执行行为树
        const status = this.root.execute();
        console.log(`行为树执行结果: ${status}\n`);
    }
    
    private simulateEnvironmentChanges(): void {
        // 模拟环境随机变化
        this.playerInSight = Math.random() > 0.5;
        this.playerInRange = this.playerInSight && Math.random() > 0.3;
        
        // 模拟受到伤害
        if (Math.random() > 0.9) {
            this.health -= 10;
            console.log(`💥 怪物受到伤害，生命值: ${this.health}`);
        }
    }
    
    // 设置怪物状态的方法
    setPlayerInSight(inSight: boolean): void {
        this.playerInSight = inSight;
    }
    
    setPlayerInRange(inRange: boolean): void {

### AI协作小技巧

##### 技巧1：让AI做单一的目标（应该做什么，越少越好）

* 只提供跟当前任务直接相关的代码和说明。**时刻思考"这句话告诉它、有什么意义"**
* 一个对话只做一个任务（功能单一、实现路径明确）
* **“单一目标”**不局限于一个函数，可以是”**有唯一实现路径**的功能”
  * 先用plan mode 让AI设计方案，review后没问题再开始执行
  * **唯一实现路径**：是指当需求、约束、边界条件都明确后，只有一种合理的方式去实现
  * **为什么需要“唯一实现路径”**？ 因为当前模块性能要求高。也就意味着我不仅需要从甲地到乙地，我还需要要求“坐明天早上八点的飞机”
  * **唯一路径到什么程度**？到我不在乎怎么达成的程度。比如具体它用的哪个接口去copy文件，比如“待拷贝文件列表“的dump格式，比如”坐哪个航空公司的航班“，这些我都不关心了，无论怎么实现都不会影响到我最终的线上效果了。

|                          ❌ 错误用法                          |                          ✅ 正确用法                          |
| :----------------------------------------------------------: | :----------------------------------------------------------: |
| 把A目录文件拷贝到B目录 ❌ 没有说明 “A目录在实时变化” ➡️ AI 会直接暴力拷贝 ❌ 没有说明 “HDFS 扫描、判断文件exist” 很耗时。➡️ AI 会每次都重新扫描 & 判断文件是否存在于B，性能较差 | **背景：**我每分钟批量产出文件到HDFS的A目录。需要将数据从A目录，实时拷贝到B目录。 **限制：**避免频繁对目录进行扫描/判断文件存在。 **新文件添加：**addFile(string[])接口，我会在每次有新文件产出时，调用这个接口 **文件异步拷贝：**addFile里将文件名记录下来后，立即返回。异步拷贝文件。 **待拷贝文件列表的管理：**要考虑重启情况，所以文件列表应该用hdfs文件系统来管理 如果A->B拷贝失败的话，最多重试3次。 |

##### 技巧2：给AI足够的约束（不应该做什么，越详细越好）

* 约束应覆盖关键点，而非事无巨细。重点关注：**必须做什么（功能）、不能做什么（限制）、边界条件（异常处理）、非功能需求（性能、兼容性）**。对于那些可以灵活选择的实现细节（比如具体算法），可以留给AI发挥。
* 自问：这个功能最核心的输入输出是什么？有哪些边界条件（重启、失败、并发）？性能要求如何？

##### 技巧3：AI 辅助拆任务

* **原理：** AI 是概率机，而且概率错误会累积——连续预测链越长，累积偏差越大。因此缩小任务空间，就能减少概率累积错误。也就是 **将复杂任务拆成多个小任务**、让AI逐个完成。

* 拆任务的过程，是我将“模糊需求”完善为“确定性需求”的过程。比如说“要给系统加个缓存”。直接让AI 设计这个工作，大概率也不太理想，因为此时这个工作叠加了太多不确定因素。

* **正确姿势是分步引导：**

|                       阶段                       |    工具     |                             例子                             |
| :----------------------------------------------: | :---------: | :----------------------------------------------------------: |
| 把模糊 idea 聊清楚 （**人类主导**） **70% 时间** |   Gemini    | 缓存设计一般都需要考虑什么？ 直接问 AI • 要缓存什么数据 → 人类问同事、看监控、整理数据格式、数据规模 • 用什么来存储 → 向 AI 学习，常见的缓存类型。让AI整理内网常见的缓存方式 • 缓存多久 → 业务评估 • 怎么失效 → 结合业务需求 + 向 AI 提问技术细节，确定失效策略 • 缓存挂了怎么办？ |
|        确定计划（**人+AI**） **20% 时间**        | Claude Code | 拆成可执行的任务后，开plan模式 开始设计实现步骤  1. 封装的 Redis 读写方法（kv格式、序列化方式、过期时间） 2. 在xxx详情接口中，查库前先查缓存，未命中再查找。查找后加入缓存。缓存查找超时100ms 3. 添加缓存命中率监控指标 |
|  把计划变成代码（**AI 主导**） **我的10% 时间**  | Claude Code |                                                              |

- 为什么用两个不同的AI？Gemini 由Google出品，具备强大搜索优势。因此Gemini更适合做发散讨论。
- Claude Code 在代码生成上更强。

### 经验教训

* 客户端永远不要自己走逻辑调用服务器暴露的主动接口，所有情况需要经过服务器校验
* c#的类与ts的类不要去stringfy，既不能输出，如果其中有函数的话又可能会循环引用抛出异常，导致代码进行不下去
* 一定要仔细校验所有接口取到数据的存在与否，不要过于相信取出来的数据就直接用来赋值，可能会导致代码报错卡死，一定一定要记得判空
* 哪些地方的代码有风险最好自己心里有点数。。。不要自行编写逻辑，合理指出不合理需求
